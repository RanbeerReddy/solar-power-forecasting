# Dataset Comparison and Selection

## Candidate Datasets

| # | Dataset | Source | Size / Resolution | Weather Variables | Regression Suitability | Notes |
|---|---------|--------|--------------------|--------------------|--------------------------|-------|
| 1 | **Solar Power Generation Data** (2 Indian PV plants) | Kaggle — `anikannal/solar-power-generation-data` | 34 days, 15-minute intervals, ~3,200+ rows per plant (inverter-level generation + plant-level sensors) | Ambient temperature, module temperature, irradiation | High — continuous target (AC/DC power), clean numeric weather features, ready for supervised regression | Widely used in published research and tutorials; well-documented; free, no special access required; two independent plants allow a train/generalisation check |

## Recommendation: Dataset #1 — Solar Power Generation Data (Kaggle, `anikannal/solar-power-generation-data`)

**Link:** https://www.kaggle.com/datasets/anikannal/solar-power-generation-data

### Why this dataset was selected

- **Clean and well-structured:** Four CSV files (two plants × generation + weather sensor pairs), consistent columns, minimal manual reformatting needed.
- **Sufficient size for a mini research project:** 15-minute readings over 34 days across 22 inverters per plant give several thousand generation rows and a directly matching weather-sensor time series — enough to train/test the four candidate models without requiring heavy compute infrastructure.
- **Includes the meteorological variables this project needs:** irradiation, ambient temperature, and module temperature are the three variables most consistently identified across the literature as dominant predictors of PV output.
- **Directly suitable for regression:** the target (AC/DC power output) is continuous, and the dataset is commonly used in exactly this way in openly published notebooks and articles, supporting reproducibility and comparability.
- **Two independent plants:** Plant 2 can optionally be used as an external validation set to check whether a model trained on Plant 1 generalises — a useful, low-cost robustness check within scope.
- **Genuinely public and license-friendly:** freely downloadable from Kaggle with no special data-sharing agreement, unlike DKASC (registration/terms) or GEFCom2014 (competition-style access), which better fits the "public dataset" and "implementable" requirements of a mini project.

### Features and Target Variable

**`Plant_1_Generation_Data.csv`** (target data, 15-min resolution):
- `DATE_TIME` — timestamp
- `PLANT_ID` — plant identifier (constant per file)
- `SOURCE_KEY` — inverter identifier
- `DC_POWER` (kW) — DC power generated by the inverter
- **`AC_POWER`** (kW) — **target variable** for this project (AC power delivered to the grid)
- `DAILY_YIELD` — cumulative energy generated that day
- `TOTAL_YIELD` — cumulative lifetime energy generated by the inverter

**`Plant_1_Weather_Sensor_Data.csv`** (feature data, 15-min resolution, one sensor array per plant):
- `DATE_TIME` — timestamp (joined with generation data)
- `AMBIENT_TEMPERATURE` (°C) — air temperature at the plant
- `MODULE_TEMPERATURE` (°C) — temperature of the PV module surface
- `IRRADIATION` — solar irradiation received (normalised units, 0–1 range in the source file)

**Engineered features (added during preprocessing, see notebook):** `hour`, `minute_of_day`, `month`, and lag features of `AC_POWER` and `IRRADIATION`, following the feature-engineering approach reported in the literature (Aouidad and Bouhelal, 2024; Ledmaoui et al., 2023).

**Target variable:** `AC_POWER` (kW), aggregated to plant level (sum across inverters) for a single time-series regression/forecasting task, or optionally modelled per-inverter.
